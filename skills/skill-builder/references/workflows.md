# Skill 工作流设计模式

## 概述

本文档提供设计 Skill 工作流的最佳实践，帮助创建清晰、有效的多步骤流程。

## 工作流类型

### 1. 顺序工作流

适用于有明确步骤顺序的任务。

```markdown
## 处理流程

1. **准备阶段**
   - 检查输入文件
   - 验证格式
   - 创建备份

2. **处理阶段**
   - 读取内容
   - 应用转换
   - 验证输出

3. **完成阶段**
   - 保存结果
   - 清理临时文件
   - 返回结果
```

### 2. 条件工作流

适用于需要根据条件选择不同路径的任务。

```markdown
## 处理流程

**如果是图片文件：**
1. 使用 PIL 读取
2. 调整尺寸
3. 保存为新格式

**如果是 PDF 文件：**
1. 使用 pdf2image 转换
2. 逐页处理
3. 合并结果

**否则：**
- 返回错误：不支持的格式
```

### 3. 迭代工作流

适用于需要重复执行直到满足条件的任务。

```markdown
## 优化流程

循环直到满足退出条件：
1. 执行当前步骤
2. 检查结果质量
3. 如果质量 < 阈值：
   - 调整参数
   - 继续循环
4. 否则：退出循环
```

## 工作流设计原则

### 清晰的分界

每个步骤应该有明确的：
- **输入**: 需要什么
- **操作**: 做什么
- **输出**: 产生什么

### 错误处理

始终在关键步骤包含错误处理：

```markdown
## 处理步骤

1. **读取文件**
   - 尝试读取
   - 如果失败：返回具体错误信息
   - 记录成功

2. **处理内容**
   - 验证输入格式
   - 如果无效：请求澄清
   - 执行处理
```

### 决策点

明确标注需要决策的地方：

```markdown
**决策点**: 选择处理方法
- 如果数据量 < 1000 条：使用内存处理
- 如果数据量 >= 1000 条：使用批处理
```

## 常见工作流模式

### 数据转换工作流

```
输入 → 验证 → 解析 → 转换 → 验证输出 → 保存
```

### 分析工作流

```
输入 → 预处理 → 分析 → 生成报告 → 格式化 → 输出
```

### 集成工作流

```
输入 → 调用 API → 处理响应 → 错误检查 → 返回结果
```

## 示例：PDF 处理工作流

```markdown
# PDF 处理工作流

## 步骤 1: 验证输入
- 检查文件是否存在
- 验证文件格式（必须是 PDF）
- 检查文件大小（警告如果 > 100MB）

## 步骤 2: 提取内容
- 使用 pdfplumber 打开
- 逐页提取文本
- 保留格式信息

## 步骤 3: 处理内容
- 清理提取的文本
- 处理特殊字符
- 保持段落结构

## 步骤 4: 输出结果
- 格式化为 Markdown
- 保存到指定路径
- 返回摘要信息
```

## 检查清单

设计工作流时，确保：

- [ ] 每个步骤都有明确的目的
- [ ] 步骤之间有清晰的过渡
- [ ] 包含错误处理路径
- [ ] 有明确的开始和结束
- [ ] 复杂度与任务匹配（不要太简单或太复杂）
